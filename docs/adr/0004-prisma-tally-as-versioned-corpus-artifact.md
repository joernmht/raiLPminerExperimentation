# 4. PRISMA tally as a versioned corpus artifact

- Status: accepted
- Date: 2026-06-22
- Deciders: Jörn Maurischat

## Context and problem statement

Paper 1 frames corpus construction as a **PRISMA flow** (identification →
screening → eligibility → included). The methods section and the PRISMA flow
diagram need exact, defensible counts ("n = …") at every transition, each with a
reason for exclusion. Hand-tallying these from logs is error-prone and drifts the
moment a record is added, deduped, snowballed, or accepted/rejected in review —
and a reviewer asking "where does n = 243 come from?" must get a reproducible
answer.

## Decision

Maintain the PRISMA tally as a **versioned corpus artifact**, the single source
of truth for every "n =" in the paper: `corpus/prisma.json` (machine record) +
`corpus/prisma.md` (human-readable) + LaTeX count macros, generated by
`corpusbuilder/prisma.py` from the corpus files. **Every stage that adds,
screens, or drops records updates it**, recording counts in/out *with exclusion
reasons*. Determinism applies (ADR-0001): same corpus files ⇒ same tally, so the
paper's methods section and flow diagram regenerate from this file rather than
being typed by hand.

## Consequences

- **Good:** every reported count is reproducible and auditable; the flow diagram
  and methods prose cannot drift from the actual corpus.
- **Good:** exclusion reasons are captured at the point of exclusion, not
  reconstructed later.
- **Bad:** the tally is a *derived* product committed to git, so it must be
  regenerated and committed whenever the corpus changes — a manual discipline
  (`python -m corpusbuilder` driver) not yet enforced by a hook or CI. A stale
  `prisma.json` would misstate the paper's counts. Mitigation: regenerate as
  part of any corpus-changing commit; consider a CI freshness check later.
